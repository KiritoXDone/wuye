# 真实支付渠道对接指南

## 1. 文档目的

本文用于指导当前项目从“本地 mock 支付参数 + mock 回调验签”切换到“真实支付渠道下单与回调验证”。

当前仓库已具备以下支付闭环基础：

- 创建支付单 `/api/v1/payments`
- 支付回调 `/api/v1/callbacks/wechatpay`、`/api/v1/callbacks/alipay`
- 回调幂等、金额校验、商户号校验、状态推进

但当前 `PaymentService` 和 `PaymentCallbackService` 仍按本地 mock 数据工作。本指南描述如何把这套骨架替换成真实渠道对接。

---

## 2. 当前代码现状

### 2.1 下单侧

当前 `PaymentService` 中：

- `WECHAT` 返回 mock `appId/timeStamp/nonceStr/package/paySign`
- `ALIPAY` 返回 mock `orderString/paySign`

也就是说：

> 当前支付参数并非来自微信/支付宝官方 SDK 或网关响应，而是本地模拟值。

### 2.2 回调侧

当前 `PaymentCallbackService` 中：

- 商户号来自 `app.payment.wechat.merchant-id` / `app.payment.alipay.merchant-id`
- 回调验签依赖本地 `callback-secret`
- 验签算法是仓库内 `PaymentSignUtils`

这是一套适合本地联调的 mock 验签机制，不是真实微信支付或支付宝通知验签方式。

---

## 3. 接入前置条件

开始前至少准备以下内容。

### 3.1 微信支付

- 商户号（mchid）
- 小程序 `appid`
- API v3 密钥
- 商户私钥
- 商户证书序列号
- 回调通知域名 / 路径白名单

### 3.2 支付宝

- 应用 `appId`
- 应用私钥
- 支付宝公钥
- 支付宝回调地址
- 签名算法配置（通常 RSA2）

### 3.3 平台侧

- HTTPS 可访问域名
- 环境变量 / Secret 管理方式
- 回调公网可达入口
- 回调日志脱敏策略

---

## 4. 推荐配置项

建议补充如下配置，不要把真实证书和密钥写在仓库里。

```yaml
app:
  payment:
    mode: mock # mock | real
    wechat:
      app-id: ${WECHAT_PAY_APP_ID:}
      merchant-id: ${WECHAT_PAY_MERCHANT_ID:}
      api-v3-key: ${WECHAT_PAY_API_V3_KEY:}
      private-key-path: ${WECHAT_PAY_PRIVATE_KEY_PATH:}
      cert-serial-no: ${WECHAT_PAY_CERT_SERIAL_NO:}
      notify-url: ${WECHAT_PAY_NOTIFY_URL:}
    alipay:
      app-id: ${ALIPAY_APP_ID:}
      merchant-id: ${ALIPAY_MERCHANT_ID:}
      private-key-path: ${ALIPAY_PRIVATE_KEY_PATH:}
      alipay-public-key-path: ${ALIPAY_PUBLIC_KEY_PATH:}
      notify-url: ${ALIPAY_NOTIFY_URL:}
```

建议保留：

- `mode=mock`：本地开发继续可跑
- `mode=real`：切到真实渠道逻辑

---

## 5. 代码改造建议

## 5.1 增加支付网关抽象

建议不要把真实渠道逻辑直接塞进 `PaymentService`。更稳妥的方式是增加网关抽象，例如：

- `PaymentGateway`
- `WechatPaymentGateway`
- `AlipayPaymentGateway`

职责拆分建议：

- `PaymentService`：业务校验、支付单创建、参数组装入口
- `PaymentGateway`：调用第三方下单 API，返回真实支付参数
- `PaymentCallbackService`：负责回调验签、幂等、金额校验、状态推进

## 5.2 下单最小闭环

建议先只打通一条真实链路，优先顺序：

1. **微信支付优先**（与当前小程序 Resident 主链路更一致）
2. 再补支付宝

真实下单时应返回：

- 微信：真实 `prepay_id` 相关参数
- 支付宝：真实 `orderString`

本轮最小闭环只需要：

- 能生成真实支付参数
- 能在前端发起支付
- 能收到成功回调

不要在第一轮一起做退款、查单补偿、分账等扩展功能。

---

## 6. 回调处理关键点

当前仓库已经具备较正确的主逻辑顺序：

1. 查支付单
2. 校验渠道一致性
3. 校验商户号
4. 校验金额
5. 幂等处理
6. 推进支付单和账单状态

切到真实渠道时，重点替换的是“**验签方式**”。

### 6.1 微信支付回调

需要替换为：

- 平台证书 / API v3 体系下的通知验签
- 必要时对资源体解密

当前本地 `PaymentSignUtils` 不能继续用于真实微信回调。

### 6.2 支付宝回调

需要替换为：

- 使用支付宝公钥对通知参数做 RSA2 验签

同样，当前本地 SHA-256 拼串签名方式只适用于 mock。

---

## 7. 幂等与一致性建议

当前项目已经有较好的支付状态保护，但切真实渠道后要额外注意：

### 7.1 重复回调

- 同一 `payOrderNo` 多次回调必须只成功入账一次
- 重复回调若交易号不同，必须拒绝或告警

### 7.2 金额校验

- 回调金额必须与本地 `payOrder.payAmount` 完全一致
- 不允许只校验“金额大于 0”或“币种正确”

### 7.3 商户号校验

- 微信/支付宝回调都必须校验商户号是否属于当前系统配置

### 7.4 回调锁

建议结合本轮新增的 Redis 回调锁能力：

- key 示例：`payment:callback:{channel}:{payOrderNo}`

这样能进一步降低高并发重复回调时的重复入账风险。

---

## 8. RabbitMQ 在支付链路中的使用建议

本轮建议 RabbitMQ 只做“支付成功事件边界”，不要让它成为支付主流程成功的前提条件。

适合放到 MQ 的后续动作：

- 支付成功后发券
- 电子凭证生成
- 发票任务通知
- 导出/报表异步任务

不适合放到 MQ 的当前关键动作：

- 支付成功本身的账单入账
- 支付单状态推进

也就是说：

> **先本地事务保证支付成功主结果，再异步发布支付成功事件。**

---

## 9. 验收步骤

建议按以下顺序验收。

### 9.1 配置验收

- `mode=mock` 时现有本地链路仍可跑
- `mode=real` 时配置能够正常加载
- 缺少关键证书/密钥时应给出明确错误

### 9.2 下单验收

- `/api/v1/payments` 返回真实渠道参数
- 前端拿到的参数可直接发起微信/支付宝支付

### 9.3 回调验收

- 渠道成功通知能通过验签
- 账单状态更新为 `PAID`
- 支付单状态更新为 `SUCCESS`
- 重复回调幂等

### 9.4 异常验收

- 错签回调被拒绝
- 金额不一致被拒绝
- 商户号不一致被拒绝
- 重复回调但交易号不一致被拒绝

---

## 10. 日志与安全要求

必须注意以下事项：

- 不要在日志里打印私钥、API v3 Key、支付宝私钥
- 不要完整记录原始回调敏感内容
- 可以记录必要的：`payOrderNo`、渠道、交易号、回调结果、错误码

建议对日志做字段脱敏。

---

## 11. 推荐最小落地顺序

建议按下面顺序实施：

1. 增加 `mode=mock/real` 配置
2. 增加支付网关抽象
3. 先接真实微信支付下单
4. 再接真实微信回调验签
5. 验收 Resident 小程序支付主链路
6. 最后再补支付宝

---

## 12. 本轮与后续边界

本轮建议完成：

- 配置设计
- 网关抽象设计
- 回调验签替换方案
- Redis / RabbitMQ 在支付链路中的接入边界说明

本轮不建议一起做：

- 退款
- 对账单下载与自动对账
- 补单任务
- 清结算
- 分账
- 多渠道统一支付运营后台

这样可以确保你先把“真实支付闭环”作为一个可控增量落地，而不是把支付系统一次性做成大平台改造。
