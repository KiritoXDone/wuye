# 项目现状盘点与差距分析

## 1. 盘点结论

基于当前仓库中的后端模块、数据库迁移、集成测试、Web 管理端路由与页面、小程序页面与服务调用，当前项目已经**明显超过 README 中描述的最小 MVP 状态**。

当前更准确的判断是：

- **MVP 主链路已完成**：认证基础、房间绑定、物业费/水费开单、住户账单查询、支付回调幂等、管理员月报等能力已经具备可运行闭环。
- **部分 P1/P2 能力已落地**：券、导入导出、Agent 授权与报表、阶梯水价、水量预警、催缴、发票申请/审核、支付凭证、审计日志等能力已经有代码、页面或测试证据支撑。
- **当前主要差距不在“缺少业务模块”**，而在于：真实渠道与生产化配置、异步基础设施、上线级鉴权与数据范围一致性、文档同步与运维准备度。

换句话说，项目当前更像是“**本地可跑通且已向 P1/P2 扩展的版本**”，而不是“只完成了部分 MVP 骨架”。

## 3.2 部分实现 / 仍需继续收口

以下能力虽然已有代码基础，但从当前仓库证据来看，更适合判断为“**已起步但未完全生产化**”：

### 认证与渠道接入

- Resident 微信登录当前仍以本地开发桩为主
- 支付虽已具备回调与幂等，但仍以本地 mock 渠道为主

### 生产化硬化

- 数据范围与权限虽然已有模块和接口，但是否已在所有查询链路中严格统一，还需要上线前再核对
- 文档描述明显落后于当前实现，存在“仓库已扩展，但 README / 子工程 README 未同步”的问题

### 前后端一致性

- 管理端和小程序已有更多页面与服务能力，但当前公开文档仍以较早期范围描述为主
- 因此项目现状、联调范围、验收口径需要再统一一次

---

## 3.3 当前仍缺失或证据不足的部分

这里的“缺失”更准确地说是“**面向生产环境仍缺关键支撑，或当前证据不足以确认已完成**”。

### 异步与基础设施

- 主配置中未看到 Redis / RabbitMQ 的明确落地配置证据
- 支付后发券、导入导出、催缴、预警等能力虽然已出现，但事件化、异步化闭环证据不足

### 真实支付与真实认证

- 真实微信登录校验
- 真实支付商户配置、真实渠道签名/验签、查单兜底等上线级能力

### 运维与上线准备

- 监控、告警、运行时可观测性证据不足
- 多环境配置与生产化部署细节未在当前盘点中看到充分证据

### 更深层组织隔离

- 组织单元已出现，但更完整的多组织 / 多租户隔离能力仍未见充分证据

## 4. 风险判断

当前项目最大的风险不是“业务模块没做”，而是“**业务能力已经铺开，但生产化约束还未完全跟上**”。

主要风险点如下：

1. **容易低估项目成熟度**：如果只看 README，会误以为项目仍停留在基础 MVP。
2. **容易高估上线准备度**：如果只看控制器、页面和测试，又容易误判为已经接近生产就绪。
3. **文档与实现不一致**：会影响联调、验收和后续迭代判断。
4. **异步与鉴权硬化不足风险较高**：这类问题在本地演示环境中不明显，但在真实运行场景中影响最大。

## 5. 下一步建议

建议下一阶段不要继续优先横向扩新模块，而是优先做以下三件事：

### 5.1 收口已存在能力

优先把以下已在仓库中出现的能力做成更明确的闭环：

- 券与支付主流程联动
- 导入导出任务闭环
- Agent / 组织单元 / 数据范围一致性
- 催缴、发票、预警的前后端验收口径

### 5.2 补生产化硬化

- 真实微信登录（已补充独立对接文档，待按文档切真实渠道）
- 真实支付渠道对接（已补充独立对接文档，待按文档切真实渠道）
- Redis / RabbitMQ 等基础设施落地（已补最小可插拔骨架与支付成功事件边界）
- 上线级幂等、补偿、监控与配置管理

### 5.2 当前补齐结果说明

截至当前版本，5.2 已补齐以下内容：

- 后端已增加 Redis / RabbitMQ 基础依赖、配置开关与最小运行骨架
- 支付回调链路已接入 Redis 回调锁抽象与 RabbitMQ 支付成功事件发布抽象
- RabbitMQ 已补最小消费闭环：支付成功事件可被消费并落审计记录，证明不是只发不收
- 基础设施不可用时，当前实现会降级而不阻断支付主链路
- Web 管理端已增加 Agent 运行配置页，可配置 API 源、Provider、模型、超时、Token 数与密钥更新
- `真实微信登录` 与 `真实支付渠道对接` 已分别形成独立对接指南，供后续按环境与证书参数逐步切换

当前仍未在本轮直接完成的部分：

- 真实微信登录 code 换取 `openid/unionid` 的正式接入
- 真实微信/支付宝支付参数下单与正式回调验签替换
- 生产级 MQ 消费重试、死信、补偿与监控闭环

### 5.3 同步文档

- 更新 README
- 更新前端子工程 README
- 如有必要，单独补一份“当前可联调能力清单”

---

## 6. 最终判断

当前项目**不缺核心业务模块骨架**，核心差距主要体现在“**从本地可跑闭环走向真实上线闭环**”这一步。

因此，项目现阶段的正确定位应为：

> 已完成 MVP 主链路，并已落地部分 P1/P2 功能；下一阶段重点应放在能力收口、生产化硬化和文档同步，而不是继续盲目扩模块。
