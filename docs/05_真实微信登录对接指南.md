# 真实微信登录对接指南

## 1. 文档目的

本文用于指导当前项目从“本地 mock 微信登录”切换到“真实微信小程序登录 code 换取用户标识”的接入方式。

当前仓库里的 `/api/v1/auth/login/wechat` 已具备登录入口，但现阶段默认仍按本地 mock 数据工作。本指南描述的是**真实接入所需准备、配置项、代码改造点、联调步骤与验收方法**。

---

## 2. 当前代码现状

当前登录链路如下：

- 前端/小程序提交 `code`
- 后端调用 `POST /api/v1/auth/login/wechat`
- `AuthService.loginWechat()` 目前直接用 `code` 命中本地 `account_identity` 映射
- 命中后签发 JWT

也就是说，当前 `code` 在后端被当成**本地身份索引**，并没有去微信官方接口换取真实的 `openid / unionid / session_key`。

要切到真实微信登录，需要把这一步改成：

> 小程序 `wx.login()` -> 拿到临时 `code` -> 后端调用微信 `jscode2session` -> 拿到 `openid/unionid` -> 再映射本地账号体系。

---

## 3. 接入前置条件

在开始改代码前，需要先准备以下信息：

### 3.1 微信小程序侧

- 小程序 `AppID`
- 小程序 `AppSecret`
- 对应的正式环境 / 测试环境域名白名单
- 小程序登录能力已在微信公众平台开通

### 3.2 后端部署侧

- 可供微信回调/访问的 HTTPS 域名
- 配置注入方式：环境变量、K8s Secret、部署平台密钥管理等
- 日志脱敏策略，确保不打印 `AppSecret`、`session_key`

### 3.3 数据模型侧

建议复用现有 `account_identity` 表，但要明确以下字段语义：

- `provider = WECHAT_MINIAPP`
- `platform_user_id = openid`
- 如有开放平台统一身份，可额外存 `unionid`

不要继续把登录临时 `code` 存成长期身份主键。

---

## 4. 推荐配置项

建议新增如下配置，不要把真实参数写死在代码里。

```yaml
app:
  auth:
    wechat:
      mode: mock # mock | real
      app-id: ${WECHAT_APP_ID:}
      app-secret: ${WECHAT_APP_SECRET:}
      jscode2session-url: https://api.weixin.qq.com/sns/jscode2session
```

推荐说明：

- `mode=mock`：保留当前本地开发联调能力
- `mode=real`：走真实微信 `code` 换会话逻辑
- `app-secret` 必须来自环境注入，不要提交到仓库

---

## 5. 后端改造建议

## 5.1 增加微信登录客户端抽象

建议新增一个专门的客户端或集成服务，例如：

- `WechatAuthClient`
- `WechatAuthService`

职责只做一件事：

- 接收前端 `code`
- 调微信官方接口
- 返回结构化结果：`openid / unionid / sessionKey / errorCode / errorMessage`

不要把 HTTP 请求、错误解析和业务账号映射全部塞回 `AuthService` 里。

## 5.2 `AuthService.loginWechat()` 的目标形态

建议把现有逻辑调整为：

1. 如果 `mode=mock`，继续沿用现有本地映射逻辑
2. 如果 `mode=real`：
   - 调微信接口换取 `openid`
   - 用 `openid` 查本地 `account_identity`
   - 找到账号后签发 JWT
   - 若未绑定账号，按产品策略决定：拒绝登录 / 自动注册 / 进入绑定流程

当前项目更稳妥的策略是：

> **先只支持“已有映射账号登录成功”，未绑定账号返回明确错误。**

这样不会在第一阶段引入额外的自动注册、副作用绑定和身份冲突处理。

## 5.3 错误处理建议

需要单独处理以下情况：

- `code` 为空或格式错误
- 微信接口超时 / 网络失败
- 微信返回 `errcode`
- `openid` 找不到本地账号映射
- 本地账号已停用

建议对外统一成可读业务错误，例如：

- `UNAUTHORIZED`：微信登录失败或账号未绑定
- `INTEGRATION_ERROR`：调用微信接口失败

---

## 6. 前端/小程序联调方式

微信小程序侧推荐流程：

1. 调 `wx.login()` 获取临时 `code`
2. 把 `code` 提交到 `/api/v1/auth/login/wechat`
3. 后端返回现有 `LoginVO`
4. 前端无需理解 `openid`，只处理 JWT 与用户态

也就是说：

> **前端请求结构可以尽量保持不变，主要变化落在后端。**

这样能最大限度减少小程序端改动面。

---

## 7. 验收步骤

建议按以下顺序验收：

### 7.1 配置验收

- `mode=mock` 时，现有 demo code 仍可登录
- `mode=real` 时，启动配置可正常加载
- 未配置 `app-id/app-secret` 时，应在启动或调用时给出明确错误

### 7.2 正常链路验收

- 小程序 `wx.login()` 拿到真实 `code`
- 后端成功换到 `openid`
- 本地存在绑定身份时，返回正常 JWT
- `/api/v1/me/profile` 可正常获取用户信息

### 7.3 异常链路验收

- 无效 `code` 返回受控错误
- 微信接口失败时返回受控错误
- 未绑定本地账号时返回明确提示

---

## 8. 上线注意事项

### 8.1 安全

- 不要记录 `AppSecret`
- 不要把完整 `session_key` 打进日志
- 不要把小程序临时 `code` 当长期身份标识使用

### 8.2 环境隔离

- 开发环境保留 `mock`
- 测试/预发环境先切 `real`
- 生产环境再切正式微信参数

### 8.3 数据准备

正式切换前，先准备好：

- `openid -> 本地 account_id` 的映射数据
- 未绑定账号时的运营处理策略

---

## 9. 建议的最小落地顺序

推荐按下面顺序实施：

1. 增加配置项与 `mode` 开关
2. 增加 `WechatAuthClient`
3. 后端接入真实 `jscode2session`
4. 保留 `mock` 回退路径
5. 做“已有账号映射即可登录”的最小闭环
6. 最后再考虑自动注册或绑定引导

---

## 10. 本轮与后续边界

本轮建议完成：

- 配置设计
- 集成抽象
- 对接步骤文档

本轮不建议一起做：

- 自动注册新住户
- 复杂绑定引导
- 多端统一身份体系改造
- 微信手机号解密绑定

这样可以先把“真实微信登录”作为一个**受控开关能力**落下来，而不破坏现有本地 mock 联调闭环。
