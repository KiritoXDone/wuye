# 前端实施补充：Web 管理端 + 微信小程序端页面与交互规范

> 版本：V1.0  
> 目标：把“前端需要 Web 端（管理员使用）、微信小程序端（用户 / 管理员使用），保证界面现代美观”这条要求，拆成可直接实施的页面、组件、交互和视觉规范。

---

## 1. 前端总体策略

建议把前端拆成两个独立工程：

```text
frontend/
├─ admin-web/           # Web 管理端
└─ miniapp-wechat/      # 微信小程序端
```

### 1.1 为什么拆成两个工程

- Web 管理端需要复杂表格、筛选、导入导出、报表和权限菜单。
- 微信小程序更适合账单查看、支付、轻量审批、快速抄表。
- 两端复用同一套后端接口，但 UI 形态、导航模式、性能约束不同。

### 1.2 角色进入方式

- **Web 管理端**：Admin / Finance 主用，登录后按角色显示菜单。
- **微信小程序端**：Resident 主用；Agent / Admin 可在同一小程序内看到不同工作台。

## 3. 视觉与交互设计原则

### 3.1 视觉关键词

- 现代
- 清爽
- 高信息密度但不杂乱
- 金融类数据清晰可核对
- 住户端更温和，管理端更专业

### 3.2 设计 Token 建议

#### 间距
- 基础栅格：`8px`
- 卡片内边距：`16px / 20px`
- 页面边距：
  - Web：`24px`
  - 小程序：`16px`

#### 圆角
- 大卡片：`16px`
- 小卡片 / 按钮：`12px`
- 输入框：`10px`

#### 阴影
- 仅用于卡片和浮层，采用轻阴影
- 表格不建议使用过重阴影

#### 字号建议
- 页面标题：`24 / 28`
- 模块标题：`18 / 20`
- 正文：`14 / 16`
- 辅助信息：`12`
- 金额重点：`24+`

### 3.3 颜色策略

- 主色：稳重蓝 / 蓝绿系
- 成功：绿色
- 警告：橙色
- 失败 / 欠费：红色
- 中性灰：用于边框、提示、辅助信息

> 不建议一开始做过多品牌色扩展，先把“状态色”和“数据可读性”做对。

## 4. Web 管理端页面规范

## 4.1 登录页

### 布局
- 左侧：品牌语 / 项目简介 / 插画或数据背景
- 右侧：登录卡片
- 登录卡片居中，避免过多元素堆叠

### 字段
- 用户名
- 密码
- 验证码（推荐）
- 登录按钮

### 状态
- 登录中按钮禁用
- 错误提示明确落在表单附近
- 首次登录强制改密弹窗

---

## 4.2 仪表盘

### 顶部 4 张 KPI 卡片
- 本月缴费户数
- 本月实收金额
- 欠费金额
- 券抵扣金额

### 中部
- 实收金额趋势图
- 缴费率趋势图
- 各用户组排行 / 欠费排行

### 右侧 / 下部
- 快捷入口：开单、导入、抄表、发券规则
- 异步任务状态：最近导入 / 导出

### 交互重点
- 所有图表卡片支持切换账期
- 指标卡点击可跳转到明细页

---

## 4.3 账单列表页

### 筛选项
- 账期
- 费用类型
- 状态
- 小区 / 组
- 楼栋 / 单元 / 房号
- 是否已支付

### 列表列建议
- 账单编号
- 房间
- 费用类型
- 账期
- 应收金额
- 已收金额
- 状态
- 到期日
- 支付时间
- 操作

### 操作区
- 查看详情
- 作废（未支付）
- 导出
- 跳转支付记录

### 体验要求
- 固定表头
- 金额列右对齐
- 状态使用清晰标签
- 高级筛选可收起

---

## 4.4 抄表页

### 推荐布局
- 上方筛选：楼栋 / 单元 / 账期
- 中间列表：房间 + 上期读数 + 本期录入 + 照片上传
- 底部悬浮操作：批量保存 / 批量生成账单

### 交互要点
- 自动带出上期 `currReading`
- 输入完成自动计算 `usage`
- `currReading < prevReading` 时立即报错
- 支持逐行保存与批量保存

---

## 4.5 导入导出页

### 导入
- 下载模板
- 选择文件
- 上传后显示处理中状态
- 完成后显示成功条数 / 失败条数
- 支持查看行级错误

### 导出
- 条件筛选
- 提交导出
- 导出任务列表
- 文件到期时间提醒

---

## 4.6 券管理页

### 券模板页
- 模板名称
- 券类型
- 抵扣方式
- 面值 / 折扣
- 门槛金额
- 有效期
- 状态

### 发券规则页
- 规则名称
- 触发条件
- 关联模板
- 最低支付金额
- 奖励数量
- 状态

### 交互建议
- 新建 / 编辑用抽屉，不用整页跳转
- 规则启停用开关控制

## 7. 页面状态规范

每个页面都必须有以下状态：

- 加载中
- 空数据
- 错误
- 权限不足
- 正常数据

### 7.1 加载态
- Web：骨架屏 + 表格行占位
- 小程序：卡片骨架 + 顶部 loading

### 7.2 空态
- 不只写“暂无数据”
- 需要告诉用户下一步怎么做，例如：
  - “暂无待缴账单”
  - “你还没有绑定房间，先去绑定”

### 7.3 错误态
- 提供“重试”按钮
- 对支付页、导入页、导出页必须给出明确错误说明

## 10. 交互细节要求

### 10.1 账单金额展示
- 应收金额、抵扣金额、实付金额必须拆开显示
- 金额要右对齐，避免核对困难

### 10.2 支付前确认
- 账单金额
- 券抵扣金额
- 实际支付金额
- 当前房间
- 当前账期

### 10.3 批量操作反馈
- Web 端批量导入 / 批量抄表 / 批量开单必须有进度和结果反馈
- 小程序端批量保存后要给出“成功 x 条 / 失败 x 条”总结

### 10.4 权限与菜单
- 菜单不要只靠前端隐藏
- 前端根据角色控制展示，后端仍必须做权限校验

## 12. 验收标准

### Web 管理端
- 登录后菜单与角色匹配
- 账单、规则、抄表、导入导出、报表可完整跑通
- 表格、筛选、分页、空态、错误态完整

### 微信小程序住户端
- 登录后能看到绑定房间
- 能完成账单查询、选券、支付、支付结果查看
- 页面风格统一、交互顺畅

### 微信小程序轻管理端
- Agent 只能看到授权组
- 可快速查看月报与房间缴费状态
- 可完成轻量抄表录入

---

## 13. 最终落地建议

- **住户高频动作放小程序**：查账单、选券、支付
- **复杂配置放 Web**：规则、开单、导入导出、报表、授权
- **Admin 在小程序只做轻量操作**：临时查状态、快速抄表、处理简单事项
- **视觉层先统一设计 Token，再开发页面**，不要一边写页面一边各自定义样式
