# AGENTS.md — 物业管理系统 AI 协作指南

> 适用范围：仓库根目录及全部子目录。若某个子目录存在更近的 `AGENTS.md`，以更近的文件为准。
>
> 目标：让 AI coding agent 在这个仓库里可以稳定地“从需求 -> 设计 -> 编码 -> 验证 -> 文档更新”闭环推进，而不是到处猜。

## 0. 优先级与工作方式

### 指令优先级
1. 用户当前任务
2. 本文件
3. 拆分后的项目文档
4. 原始项目文档
5. 代码中的既有实现与命名习惯

若文档与现有代码冲突：
- **先保留线上/现有稳定行为**，不要在未说明迁移方案时直接推翻。
- 在交付说明中明确写出“冲突点 / 当前处理 / 推荐后续动作”。

### 输出与协作约定
- 沟通、注释、文档默认使用 **中文**。
- 代码、接口字段、数据库字段使用 **英文**。
- 所有新文件使用 **UTF-8** 编码。
- 优先给出 **最小可运行闭环**，不要一开始大面积重构。
- 默认采用 **先纵向切一条链路，再扩展** 的方式开发，而不是同时摊开所有模块。

### 每次开始任务前必须做的事
1. 先识别任务属于哪个模块：认证 / 房间 / 账单 / 支付 / 券 / 报表 / Web 管理端 / 微信小程序。
2. 打开相关文档，确认字段、状态、接口和业务口径。
3. 找仓库中已有实现、DTO、Mapper、Controller、页面组件，优先复用既有模式。
4. 再动代码。

## 1. 项目是什么

这是一个 **物业管理系统**，目标是从零构建出完整可上线的缴费与运营后台。

### 后端默认技术栈
- Java 17
- Spring Boot 3.x
- MyBatis
- MySQL 8
- Redis
- RabbitMQ
- Maven

### 前端默认形态
- **Web 管理端**：给管理员 / 财务使用，承担复杂表格、筛选、导入导出、报表、规则配置。
- **微信小程序端**：给住户使用，也承载 Agent / Admin 的轻量工作台。

### 前端默认选型（如果仓库尚未固定）
- `frontend/admin-web`：**Vue 3 + TypeScript + Vite + Element Plus + Pinia + Vue Router**
- `frontend/miniapp-wechat`：**原生微信小程序 + TypeScript**

> 除非用户明确要求，**不要擅自切换到 React / Taro / uni-app / Next.js 等另一套栈**。如果仓库已经存在前端工程，则继续沿用现有栈，不做框架迁移。

## 3. 绝对不能写错的业务真相

以下规则是本项目的“地基”，任何功能都不能违背。

### 3.1 房间是账单主体，不是账号
- `room` 是账单生成、统计、支付闭环的核心主体。
- `account` 只是查看 / 支付 / 绑定主体。
- **多个账号可以绑定同一个房间**。
- 同一房间其中一个账号支付成功后，其他已绑定账号应看到同样的已支付结果。

### 3.2 权限是两层：RBAC + 数据范围
- `Resident`：只看自己绑定房间的数据（Self）。
- `Agent`：只看被授权用户组的数据（Group）。
- `Admin`：看全局（All）。
- 如存在 `Finance`，默认是全局报表 / 导出角色，但不要在未明确设计前滥加权限。

### 3.3 计费规则不能漂移
- **物业费**：按面积计费。
- **水费**：按抄表读数计费。
- 水费用量 = `curr_reading - prev_reading`，必须 `>= 0`。
- 金额统一使用两位小数；后端使用 `BigDecimal`，禁止用 `double/float` 计算金额。

### 3.4 账单唯一性和支付一致性
- 同一 `room_id + fee_type + period_year + period_month` 默认只能有一笔有效账单。
- 已支付账单 **不能直接覆盖**；如需修正，应走“作废 / 冲正 / 差额补收”方案。
- 支付回调必须支持 **重复回调幂等**。
- 优惠券核销必须支持 **并发幂等**。

### 3.5 券系统语义不能混淆
- **支付前抵扣券**：影响实际支付金额。
- **支付后发放兑换券 / 奖励券**：在支付成功后发放。
- 不要把“券模板”“券实例”“核销记录”混成一张表解决。

### 3.6 统计口径按房间，不按账号
- 缴费户数 / 欠费户数 / 缴费率等核心指标，默认以 **房间** 为统计口径。
- 一个房间绑定多个账号，不应在报表中重复计算成多户。

## 4. 从零开发时的推荐顺序

如果用户给的是“大任务”或“从头搭建”，按下面顺序推进：

1. **工程初始化**：基础目录、环境配置、统一响应、全局异常、日志
2. **认证鉴权**：登录、JWT、RBAC、数据范围拦截
3. **房间与绑定**：`account` / `room` / `account_room`
4. **账单闭环**：物业费开单、水费抄表、账单查询、账单详情
5. **支付闭环**：支付单、渠道下单、回调、查单兜底
6. **券系统**：模板、实例、核销、支付前抵扣、支付后发券
7. **导入导出**：导入批次、异步导出、对象存储
8. **报表与审计**：管理员报表、Agent 组内报表、操作审计
9. **前端完善**：Web 管理端、微信小程序端、交互与视觉收口

### 切功能时的默认策略
- 优先实现 **Resident 小程序缴费链路**：登录 -> 我的房间 -> 账单列表 -> 账单详情 -> 支付
- 再实现 **Web 管理端最小闭环**：登录 -> 费用规则 -> 开单 -> 抄表 -> 账单查询 -> 报表
- 再做 Agent / 券 / 导入导出 / 高级报表

## 6. 编码风格与实现约束

### 6.1 后端
- 使用清晰的 DTO / Query / VO，不要让 Controller 直接暴露数据库实体。
- 业务逻辑写在 Service，SQL 写在 Mapper/XML 或注解中，但复杂 SQL 优先 XML，便于维护。
- 统一响应模型、统一错误码、统一异常处理。
- 涉及金额、状态、时间范围的字段，命名保持一致，不随意缩写。
- 对外 API 前缀保持 `/api/v1`。

### 6.2 数据库
- 主键默认 `bigint`。
- 时间字段统一：`created_at`、`updated_at`，必要时加 `paid_at`、`issued_at` 等业务时间。
- 重要状态字段使用清晰枚举值，不要混用 `0/1/2/3` 且不解释。
- 有唯一业务约束时，必须落到数据库唯一索引，不能只靠代码判断。
- 涉及支付、券、账单等强一致性模块时，优先保证正确性，再谈缓存。

### 6.3 前端
- Web 管理端：现代、简洁、专业，优先使用卡片 + 表格 + 抽屉 / 弹窗，不做花哨动画。
- 微信小程序：住户优先，流程短，按钮清晰，金额突出，避免信息过载。
- 必须覆盖 **加载态 / 空态 / 错误态 / 成功反馈**。
- 账单金额、实付金额、优惠金额必须清晰区分。
- 不得把权限控制只放在前端；前端只做展示收敛，权限以后端为准。

### 6.4 文档与注释
- 新增模块时，同步更新相关文档。
- 新增接口时，至少更新接口文档或联调文档。
- 新增表 / 字段 / 索引时，至少更新 DDL 与设计说明。
- 对复杂流程（支付、核销、异步导出），在代码附近保留必要注释。

## 7. 按模块的具体开发规则

## 7.1 认证与权限
- 登录先做最小闭环：微信小程序登录 + JWT。
- Web 管理端如需账号密码登录，可独立实现 `admin_auth` 或等价设计。
- 数据范围必须在后端落实，不允许前端通过传参自行限制范围来“假装安全”。
- Agent 是否可查看 / 管理某个用户组，以 `agent_group` 为准。

### 认证改动的完成标准
- 登录成功
- Token 校验通过
- Resident / Agent / Admin 越权测试至少各有一条
- 菜单和接口权限一致

## 7.2 房间与绑定
- 房间查询以“我绑定的房间”为主入口。
- 绑定状态要明确，如 `PENDING / ACTIVE / INACTIVE`。
- 绑定关系是核心关系，修改前先看是否影响账单共享与历史查询。

### 绝对不要
- 把“房间归属”偷换成“账号归属”
- 因为一个账号解绑，就删除房间历史账单

## 7.3 账单
- 账单至少拆成 `bill` + `bill_line` 两层。
- 物业费开单依赖面积和有效计费规则。
- 水费开单依赖抄表记录和有效水费规则。
- 开单前先做唯一性检查；真正兜底靠数据库唯一索引。

### 账单模块完成标准
- Resident 能按房间查账单
- Admin 能开物业费账单
- Admin 能录抄表并生成水费账单
- 同一房间同账期不会重复开有效账单

## 7.4 支付
- 先把 `pay_order`、`pay_transaction` 设计稳定，再接渠道。
- 创建支付单时必须有幂等键，防止重复点击。
- 回调处理要做：验签 / 金额校验 / 商户号校验 / 幂等处理 / 状态推进。
- 回调成功后，账单状态、支付单状态、支付流水必须一致。
- 对查单兜底流程预留接口或任务。

### 绝对不要
- 在 Controller 里直接写完整支付流程
- 把“支付成功”仅靠前端返回值判断
- 把渠道返回原文不加筛选直接落日志

## 7.5 券系统
- 至少区分：`coupon_template`、`coupon_instance`、`coupon_redemption`。
- 抵扣校验必须看：状态、有效期、适用费种、最小门槛、是否可叠加。
- 核销时优先使用事务 + 行锁 / 乐观锁 + Redis 锁组合，保证并发安全。
- 支付成功发券优先走事件化 / 异步化，而不是同步塞进支付主流程。

## 7.6 导入导出
- 导入按“批次 + 行错误”设计，不要导入失败就整批无痕消失。
- 导出采用异步任务模型，避免大查询阻塞请求线程。
- 文件存储路径、对象存储 key、过期清理策略要可追溯。

## 7.7 报表
- 管理员报表看全局。
- Agent 报表只看授权组。
- 报表口径默认按房间统计。
- 做统计查询时优先保证口径正确，再考虑缓存和预聚合。

## 7.8 Web 管理端
- 首页至少要有：应收 / 实收 / 缴费率 / 欠费户数 四类核心 KPI。
- 账单列表必须支持筛选、分页、状态展示、导出。
- 抄表页必须支持按楼栋 / 单元筛选和批量录入。
- 券管理页至少有：模板管理、发券规则、使用记录。
- UI 风格：留白充足、圆角统一、信息层级清晰、少而准的强调色。

## 7.9 微信小程序端
- Resident 首页要优先展示：我的房间、待缴账单、最近账单、我的券。
- 支付页要明确展示：应付、优惠、实付。
- 支付结果页必须有成功 / 失败 / 处理中三态。
- Agent / Admin 轻工作台可提供：用户组列表、组内报表、轻量抄表。

## 8. 事务、锁、缓存、消息队列

### 必须加事务的地方
- 创建支付单并冻结券
- 处理支付成功回调
- 券核销
- 账单生成（尤其批量开单时的单房间写入）

### 必须考虑锁 / 幂等的地方
- 重复点击支付
- 重复支付回调
- 并发核销同一张券
- 同账期重复开单

### Redis 最小使用策略
- 幂等键：`pay:idem:{idempotencyKey}`
- 券锁：`coupon:lock:{couponInstanceId}`
- 热点账单缓存：`bill:detail:{billId}`

### RabbitMQ 最小使用策略
- 支付成功后发券
- 导入 / 导出异步化
- 消费成功再 ack，失败进入重试或死信

## 10. Definition of Done

- 代码可编译
- 关键路径可运行
- 异常路径可解释
- 涉及的 DTO / VO / Mapper / Service / Controller 成对齐

- 不留下“后面再补”的关键业务洞

- DDL 已更新
- 索引 / 唯一约束已说明
- 旧数据兼容性已说明

### 前端需求完成，至少满足
- 页面能进入
- 接口已接通
- 加载 / 空 / 错 / 成功反馈齐全
- 移动端点击区、Web 端表格交互可用
- 权限菜单与后端能力一致

### 支付 / 券 / 报表这类高风险功能完成，至少满足
- 幂等已考虑
- 并发已考虑
- 审计 / 日志已考虑
- 状态机已闭环
- 报表口径与业务规则一致

## 11. 明确禁止事项

- 不要把金额计算写成 `double`。
- 不要绕过数据库唯一约束，只在 Java 代码里做“软判断”。
- 不要在未设计迁移方案时修改核心表主语义（如把账单主体从 `room` 改成 `account`）。
- 不要直接覆盖已支付账单金额。
- 不要把权限校验只写在前端。
- 不要把大段业务逻辑塞进 Controller / Page。
- 不要把密钥、商户号私钥、支付证书写进代码仓库。
- 不要新增与当前栈冲突的大型依赖，除非用户明确批准。
- 不要生成一堆临时文件丢在仓库根目录；研究资料放 `docs/` 或 `docs/research/`。
- 不要为了“看起来快”而跳过测试、幂等和状态校验。

## 12. 推荐的任务响应格式（给 AI agent 自己用）

每次接到任务，优先按下面结构思考和输出：

1. **任务理解**：一句话说明要做什么
2. **影响范围**：会动哪些模块 / 文件
3. **实现方案**：最小可行闭环是什么
4. **关键风险**：权限、幂等、状态、口径、兼容性
5. **改动结果**：完成了什么
6. **验证结果**：跑了什么、过了什么
7. **剩余事项**：还有哪些非阻塞项可后续补

## 13. 两个好用的默认策略

### 策略 A：做新功能时
- 先完成后端最小接口 + 数据结构
- 再接一个最小页面 / 最小调用链路
- 再补异常态、权限、测试、文档

### 策略 B：做问题修复时
- 先定位是文档问题、口径问题、状态机问题，还是纯代码 bug
- 修复时尽量最小范围
- 如果改动会影响账单 / 支付 / 券历史数据，必须说明影响面

## 14. 可以直接复用的实现偏好

### 金额处理
- 统一用 `BigDecimal`
- 统一两位小数
- 统一在服务层完成金额计算与舍入

### 状态命名
优先使用可读枚举值，例如：
- `PENDING`
- `ACTIVE`
- `INACTIVE`
- `ISSUED`
- `PAID`
- `FAILED`
- `USED`
- `EXPIRED`

### Web UI 偏好
- 卡片式仪表盘
- 清晰筛选区
- 表格操作列收敛
- 抽屉优于多层弹窗
- 现代但克制，不做炫技动画

### 小程序 UI 偏好
- 关键金额大字号
- CTA 明确
- 账单卡片信息不超过 5~6 个主字段
- 支付链路尽量少跳转
