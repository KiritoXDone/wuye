## 1. 当前项目的上线基线

这是一个 **物业管理系统**，当前不是纯“从零设计稿”，而是已经有了明确的**上线冻结项**。

### 当前冻结业务口径
- **账单主体永远是房间（room）**，不是账号。
- **物业费按面积计费，按年开单，按年缴纳**。
- **水费按月抄表，录入系统后立即生成当月账单**。
- **账单按费用类型分单**，不要把物业费和水费混成一张总账单。
- **报表默认按房间统计**，不是按账号统计。

### 当前冻结前端口径
- **Web 管理端**：`React + Tailwind CSS + TypeScript`
- 若仓库尚未固定路由方案，默认：`React Router`
- 构建产物由 **Nginx** 发布
- 需要支持静态包快速回滚

### 当前冻结后端基线
- Java 17
- Spring Boot 3.x
- MyBatis
- MySQL 8
- Redis
- RabbitMQ
- Maven

> 除非用户明确要求，**不要擅自切换到 Vue / Next.js / Taro / uni-app / 另一套支付链路 / 另一套数据库模型**。如果仓库已经存在对应工程，则继续沿用现有栈，不做框架迁移。

## 3. 绝对不能写错的业务真相

以下规则是本项目的地基，任何功能都不能违背。

### 3.1 房间是账单主体，不是账号
- `room` 是账单生成、统计、支付闭环的核心主体。
- `account` 只是查看 / 支付 / 绑定主体。
- **多个账号可以绑定同一个房间**。
- 同一房间任一绑定账号支付成功后，其他绑定账号应看到同样的已支付结果。

### 3.2 权限是两层：RBAC + 数据范围
- `Resident`：只看自己绑定房间的数据（Self）
- `Agent`：只看被授权用户组的数据（Group）
- `Admin`：看全局（All）
- `Finance`：如存在，默认是报表 / 导出 / 对账角色，但不要在未明确设计前滥加权限

### 3.3 物业费不是按月，而是按年
- 物业费仍然按面积计费。
- 但当前上线口径是：**每房间每自然年只生成一张物业费账单**。
- 默认支付方式是：**一年一交**。
- 没有用户明确要求时，不要再把当前实现写回“月度物业费”。
- 物业费账单应能明确表示服务周期，如：`2026-01-01 ~ 2026-12-31`。

### 3.4 水费按月抄表，录入即出账
- 水费用量 = `curr_reading - prev_reading`，必须 `>= 0`。
- 同一房间同一自然月只能有一条有效抄表记录。
- 管理员录入抄表后，应在**同一业务闭环**中生成当月水费账单和明细。
- 不要把“先录抄表、后人工再去单独点一次生成账单”当成默认生产口径，除非用户明确要求分两步。

### 3.5 账单模型必须支持年费和月费并存
- 账单至少拆成 `bill` + `bill_line` 两层。
- `bill` 建议显式支持：
  - `cycle_type`：如 `YEAR` / `MONTH`
  - `service_period_start`
  - `service_period_end`
- 账单金额统一使用两位小数；后端统一用 `BigDecimal`，禁止用 `double/float` 计算金额。

### 3.6 唯一约束不能漂移
- 物业费账单唯一约束：`UNIQUE(room_id, fee_type, period_year)`
- 水费账单唯一约束：`UNIQUE(room_id, fee_type, period_year, period_month)`
- 抄表唯一约束：`UNIQUE(room_id, period_year, period_month)`
- 代码里的“先查一下有没有”只能做前置校验；真正兜底必须落到数据库唯一索引。

### 3.7 已支付账单不能直接覆盖
- 已支付账单禁止直接改金额、改期次、改主键语义。
- 如需修正，应走：
  - 作废
  - 冲正
  - 差额补收
  - 重新出单
- 支付回调必须支持**重复回调幂等**。

### 3.8 券系统语义不能混淆
- **支付前抵扣券**：影响实付金额。
- **支付后发放奖励券 / 兑换券**：支付成功后发放。
- 不要把“券模板”“券实例”“核销记录”混成一张表。
- 优惠券核销必须支持并发幂等。

### 3.9 统计口径按房间，不按账号
- 缴费户数 / 欠费户数 / 缴费率等核心指标，默认以 **房间** 为口径。
- 一个房间绑定多个账号，报表中不能重复算成多户。

### 3.10 上线链路必须可审计、可对账、可回滚
- 费用规则变更、抄表录入、账单生成、账单作废、发券、冲正、导入导出、权限调整，都应可审计。
- 支付链路至少要有 **T+1 对账**。
- 管理端静态包发布必须有回滚方案。
- 涉及账单结构变更的数据库脚本，发布前必须演练。

## 5. 开发推进顺序（以上线闭环为导向）

如果用户给的是“大任务”或“从头搭建”，按下面顺序推进：

1. **工程初始化**：环境配置、统一响应、全局异常、日志、DB migration
2. **认证鉴权**：登录、JWT、RBAC、数据范围拦截
3. **房间与绑定**：`account` / `room` / `account_room`
4. **年度物业费闭环**：年费规则、按年开单、账单查询
5. **月度水费闭环**：抄表录入、录入即出账、账单查询
6. **支付闭环**：支付单、渠道下单、回调、查单兜底、T+1 对账
7. **报表与导入导出**：年费报表、月水费报表、导入导出
8. **券系统**：模板、实例、核销、支付前抵扣、支付后发券
9. **审计与上线准备**：操作审计、监控、备份恢复、回滚演练
10. **前端完善**：Web 管理端收口、居民端联动、交互与权限收口

### 切功能时的默认策略
- 优先实现 **Web 管理端最小闭环**：登录 -> 年度物业费 -> 月度抄表 -> 账单列表 -> 报表
- 若仓库包含 Resident 端，再实现：登录 -> 我的房间 -> 账单列表 -> 账单详情 -> 支付
- 再做 Agent / 券 / 导入导出 / 高级报表 / 运维增强

### 生产问题修复默认策略
- 先确认是：文档口径问题 / 业务规则问题 / 状态机问题 / 数据问题 / 纯代码 bug
- 尽量做**最小范围修复**
- 若可能影响历史数据，必须补：
  - 影响分析
  - 数据修复脚本
  - 回滚方案
  - 验证方式

## 6. 编码风格与实现约束

### 6.1 后端
- 使用清晰的 DTO / Query / VO，不要让 Controller 直接暴露数据库实体。
- 业务逻辑写在 Service；复杂 SQL 优先写 Mapper XML，便于维护和审查。
- 统一响应模型、统一错误码、统一异常处理。
- 对外 API 前缀保持 `/api/v1`。
- 涉及金额、状态、时间范围、周期类型的字段，命名要稳定，不随意缩写。
- 重要表结构改动优先走 Flyway / Liquibase / 规范化 SQL 脚本，不要手改后忘记沉淀。

### 6.2 数据库
- 主键默认 `bigint`。
- 时间字段统一：`created_at`、`updated_at`，业务时间按需加 `issued_at`、`paid_at`、`reading_at`、`service_period_start`、`service_period_end`。
- 重要状态字段使用可读枚举，不要只留 `0/1/2/3` 而不解释。
- 强业务约束必须有数据库索引或唯一约束，不能只靠 Java 代码判断。
- 支付、券、账单、抄表等强一致性模块，优先保证正确性，再谈缓存与性能。

### 6.3 前端
- Web 管理端：现代、克制、专业，优先使用卡片 + 表格 + 抽屉 / 弹窗。
- 当前默认风格基线：**React + Tailwind**。
- 未获批准前，不要自行引入与当前栈强耦合的大型 UI 体系替换现有实现。
- 必须覆盖：**加载态 / 空态 / 错误态 / 成功反馈 / 权限无权态**。
- 账单应收、优惠、实付、账期、服务周期必须清晰区分。
- 权限控制以后端为准，前端只做展示收口，不能假装安全。

### 6.4 文档与注释
- 新增模块时，同步更新相关文档。
- 新增接口时，至少更新接口文档或联调文档。
- 新增表 / 字段 / 索引时，至少更新 DDL 与设计说明。
- 改了上线行为、发布流程、回滚方式、对账方式，必须同步更新上线文档。
- 对复杂流程（支付、核销、抄表出账、异步导出、回滚），在代码附近保留必要注释。

## 7. 模块开发规则

## 7.1 认证与权限
- 登录先做最小闭环：JWT + 当前用户上下文。
- Web 管理端如需账号密码登录，可独立实现 `admin_auth` 或等价设计。
- 数据范围必须在后端落实，不允许前端通过传参假装做安全控制。
- Agent 是否可看某个用户组，以 `agent_group` 为准。

### 认证改动完成标准
- 登录成功
- Token 校验通过
- Resident / Agent / Admin 越权测试至少各有一条
- 菜单权限与接口权限一致

## 7.2 房间与绑定
- 房间查询以“我绑定的房间”为主入口。
- 绑定状态要明确，如 `PENDING / ACTIVE / INACTIVE`。
- 修改绑定关系前，先评估是否影响账单共享、历史支付结果和报表口径。

### 绝对不要
- 把“房间归属”偷换成“账号归属”
- 因为一个账号解绑，就删除房间历史账单

## 7.3 物业费账单
- 物业费按面积计费，但**按年开单**。
- 建议保留单独子服务，如：
  - `PropertyFeeRuleService`
  - `PropertyYearlyBillGenerateService`
  - `PropertyBillQueryService`
- 年度开单前先做唯一性检查；真正兜底靠数据库唯一索引。
- 年度账单必须能解释计算依据：面积、单价、服务周期、应收金额。

### 物业费模块完成标准
- Admin 能生成年度物业费账单
- Resident 能按房间查到年度物业费账单
- 同一房间同一年不会重复生成有效物业费账单
- 账单详情可解释金额来源和服务周期

## 7.4 水费抄表与账单
- 水费按月抄表。
- 抄表录入后默认立即生成账单，不要把出账留给人工补操作。
- 建议保留单独子服务，如：
  - `WaterReadingService`
  - `WaterBillGenerateService`
  - `WaterBillQueryService`
- 必须校验：
  - `curr_reading >= prev_reading`
  - 同房间同年月无重复抄表
  - 水费规则有效
- 如支持照片证据，文件路径和对象存储 key 必须可追溯。

### 水费模块完成标准
- Admin 能录入月度抄表
- 录入后能自动生成当月水费账单
- 同一房间同一自然月无重复抄表、无重复有效水费账单
- Resident 能按房间查到月度水费账单

## 7.5 账单查询与详情
- 账单至少拆成 `bill` + `bill_line`。
- 前台与后台都必须能清晰区分：
  - `fee_type`
  - `cycle_type`
  - `period_year`
  - `period_month`（如适用）
  - `service_period_start`
  - `service_period_end`
  - `status`
  - `amount_due`
  - `amount_discount`
  - `amount_paid`
- 账单详情必须可解释，不允许只给一个总金额。

## 7.6 支付
- 先把 `pay_order`、`pay_transaction` 设计稳定，再接渠道。
- 创建支付单时必须有幂等键，防止重复点击。
- 回调处理要做：验签 / 金额校验 / 商户号校验 / 幂等处理 / 状态推进。
- 回调成功后，账单状态、支付单状态、支付流水必须一致。
- 必须为查单兜底和 T+1 对账预留接口或任务。

### 绝对不要
- 在 Controller 里直接写完整支付流程
- 把“支付成功”仅靠前端返回值判断
- 把渠道返回原文不加筛选直接落日志

## 7.7 券系统
- 至少区分：`coupon_template`、`coupon_instance`、`coupon_redemption`。
- 抵扣校验必须看：状态、有效期、适用费种、最小门槛、是否可叠加。
- 核销时优先使用事务 + 行锁 / 乐观锁 + Redis 锁组合，保证并发安全。
- 支付成功发券优先走事件化 / 异步化，而不是同步塞进支付主流程。

## 7.8 报表
- 管理员报表看全局。
- Agent 报表只看授权组。
- 报表口径默认按房间统计。
- 当前至少要支持：
  - 年度物业费报表
  - 月度水费报表
  - 应收 / 实收 / 欠费 / 缴费率
- 做统计查询时优先保证口径正确，再考虑缓存和预聚合。

## 7.9 导入导出
- 导入按“批次 + 行错误”设计，不要导入失败就整批无痕消失。
- 导出采用异步任务模型，避免大查询阻塞请求线程。
- 文件路径、对象存储 key、过期清理策略要可追溯。

## 7.10 审计
- 以下动作默认必须审计：
  - Agent 授权变更
  - 费用规则变更
  - 年度物业费开单
  - 月度抄表录入
  - 账单作废 / 冲正
  - 券模板变更

## 9. 事务、锁、缓存、消息队列

### 必须加事务的地方
- 创建年度物业费账单（尤其批量开单时按房间写入）
- 录入月度抄表并生成水费账单
- 创建支付单并冻结券
- 处理支付成功回调
- 券核销
- 冲正 / 差额补收 / 作废重建

### 必须考虑锁 / 幂等的地方
- 同年度重复生成物业费账单
- 同月份重复录入抄表
- 同账单重复点击支付
- 重复支付回调
- 并发核销同一张券
- 导入任务重复提交

### Redis 最小使用策略
- 幂等键：`pay:idem:{idempotencyKey}`
- 券锁：`coupon:lock:{couponInstanceId}`
- 热点账单缓存：`bill:detail:{billId}`
- 月度抄表去重保护（可选）：`reading:lock:{roomId}:{periodYear}:{periodMonth}`

### RabbitMQ 最小使用策略
- 支付成功后发券
- 导入 / 导出异步化
- 对账任务投递
- 消费成功再 ack，失败进入重试或死信

## 10. 发布、回滚、对账、监控

### 发布要求
- 发布前演练账单结构相关脚本，特别是 `bill` 的周期字段和服务周期字段变更。
- 管理端静态资源发布到 Nginx 前，必须保留上一版本构建产物以便回滚。
- 前后端涉及同一口径变更时，必须确认发布顺序和兼容窗口。

### 回滚要求
- 前端至少支持静态包快速回滚。
- 后端若涉及数据库变更，必须给出：
  - 是否可逆
  - 回滚脚本
  - 数据影响说明
- 无回滚方案的高风险改动不要直接落生产。

### 对账要求
- 支付链路至少执行 **T+1 对账**。
- 对账差异必须能定位到：账单、支付单、渠道流水、券状态。
- 对账修复动作必须留审计。

### 监控要求
建议至少监控：
- API QPS / RT / 错误率
- 支付回调成功率
- MQ 堆积量
- 导入导出任务成功率
- MySQL 慢 SQL
- Redis 命中率
- Nginx 5xx / 静态资源发布异常

### 备份与恢复
- MySQL：全量 + binlog
- 对象存储：开启版本或生命周期管理
- 导出文件：设置过期清理
- 上线前至少做一次恢复演练

### 每次改动后的最低验证要求
- 改后端接口：至少跑相关单测 / 集成测试，并自查 1 条正常流 + 1 条异常流。
- 改数据库：确认索引、唯一约束、默认值、历史兼容性、回滚方式。

- Resident / Agent / Admin 的数据范围隔离正确

## 13. 明确禁止事项

- 不要把金额计算写成 `double`。
- 不要绕过数据库唯一约束，只在 Java 代码里做“软判断”。
- 不要在未设计迁移方案时修改核心表主语义。
- 不要把账单主体从 `room` 改成 `account`。
- 不要把当前生产口径的年度物业费改回月度物业费。
- 不要把“水费录入即出账”改成默认人工补出账流程，除非用户明确要求。
- 不要直接覆盖已支付账单金额。
- 不要把权限校验只写在前端。
- 不要把大段业务逻辑塞进 Controller / Page。
- 不要把密钥、商户号私钥、支付证书写进代码仓库。
- 不要为了图省事跳过测试、幂等、对账、审计和回滚说明。
- 不要未经批准把 React + Tailwind 管理端改成另一套前端框架。
- 不要生成一堆临时文件丢在仓库根目录；研究资料放 `docs/` 或 `docs/research/`。

## 14. 推荐的任务响应格式（给 AI agent 自己用）

每次接到任务，优先按下面结构思考和输出：

1. **任务理解**：一句话说明要做什么
2. **影响范围**：会动哪些模块 / 文件 / 数据 / 发布链路
3. **实现方案**：最小可行闭环是什么
4. **关键风险**：权限、幂等、状态、口径、兼容性、上线风险
5. **改动结果**：完成了什么
6. **验证结果**：跑了什么、过了什么
7. **剩余事项**：还有哪些非阻塞项可后续补

如果任务涉及 SQL、接口、页面、上线中的任意两项以上，交付说明里要额外补：
- 文档同步情况
- 是否需要迁移脚本
- 是否影响回滚
